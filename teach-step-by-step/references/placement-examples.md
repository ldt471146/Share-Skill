# Placement Examples

用这些例子判断一条内容应该放到哪里。

## 放进 rules/

句子形态像“你必须做 X”或“禁止做 Y”。

示例：

- 每次教学只能问一个检查问题。
- 用户没有确认理解前，不要连续推进多个新概念。

原因：这些是所有教学任务都要遵守的稳定约束。

## 放进 workflows/

句子形态像“第 1 步、第 2 步、第 3 步”。

示例：

- 先阅读资料结构。
- 再拆成小单元。
- 然后逐个讲解并提问。

原因：这些是某类任务的执行顺序。

## 放进 references/

句子形态像“举例说明”“背景解释”“小心这个坑”。

示例：

- 用户说“还是不懂”时，不要重复原定义；换一个更小的例子。
- 如果一条教学经验只来自某次对话，先不要写进规则，除非它可重复、代价高、以后还会遇到。
- “慢慢教我这个文件”“仔细拆解这篇文章”是同一任务的不同话术：话术放 `description`，流程合并到 `Common Tasks` 的 `workflows/teach-source.md`。
- 多个 skill 都需要同一条稳定规则时，不要复制到每个 `SKILL.md`；放到共享规则，再让各 skill 的 Always Read 指向它。

原因：这些内容帮助理解，但不应该让导航文件变长。

## 放进 scripts/

内容需要反复执行，而且靠人工容易漏。

示例：

- 检查 `SKILL.md` frontmatter 是否有效。
- 检查 `Common Tasks` 引用的文件是否存在。

原因：确定性检查交给脚本比交给记忆更可靠。
